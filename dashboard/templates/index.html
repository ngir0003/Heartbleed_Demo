The Heartbleed Bug

A missing bounds check in OpenSSL 1.0.1f lets any attacker silently read 64 KB of server memory per request — leaking passwords, private keys, and session tokens without leaving a trace.

Attack Flow

From Login to Session Takeover

1Step 01
Victim logs in
SecureBank sets a session= cookie. The HTTPS request passes through OpenSSL's heap buffers.
2Step 02
Attacker connects
No credentials needed — the heartbeat extension is available to any peer before authentication.
3Step 03
Malformed heartbeat
Packet carries 1 byte but claims 0x4000 (16,384). OpenSSL's memcpy uses the attacker-supplied length — no bounds check.
4Step 04
16 KB of heap leaked
The response echoes 16,383 bytes of adjacent heap data — which may include the victim's session cookie.
5Step 05
Session hijacked
Attacker pastes the stolen cookie into their browser. Full account access granted — no password required.

Guided Demo

Walkthrough — Step by Step

# Step Who Action What Happens
1 Open the bank Victim Go to https://localhost:4001, accept the certificate warning, log in as alice / password123. Server issues a Set-Cookie: session=… token. That token flows through OpenSSL's heap on every request.
2 Open the console Attacker In a new tab go to http://localhost:4002. Target (server:443) is pre-filled. Click Run Heartbleed Exploit. The tool logs the victim in automatically to seed the heap, then fires up to 20 malformed heartbeat probes.
3 Read the dump Attacker Watch the hex dump appear below the result panel. Look for Cookie: session= in the ASCII column on the right. Each probe leaks up to 16,384 bytes of raw heap. The token appears in plaintext — no decryption needed.
4 Copy the token Attacker When the green Token Found badge appears, copy the 32-character hex token shown below it. The attacker now holds the victim's live session identifier — obtained without any password.
5 Hijack the session Attacker DevTools → Application → Cookies (Chrome) or Storage → Cookies (Firefox). Add cookie: session = stolen token, domain localhost. Navigate to https://localhost:4001/dashboard. The server accepts the token and grants full account access — no password ever entered.
⚠ Why this works — The TLS heartbeat extension (RFC 6520) is negotiated before authentication. OpenSSL 1.0.1f trusts the attacker-supplied payload length with no bounds check, so memcpy copies adjacent heap memory into the response. The victim's Cookie: header passes through those heap buffers on every HTTPS request — making it recoverable without any decryption.

Root Cause Analysis

Vulnerable Code Path — OpenSSL t1_lib.c

  openssl / ssl / t1_lib.c VULNERABLE
1n2s(p, payload);
2pl = p;
  
3bp = OPENSSL_malloc(1 + 2 + payload + padding);
4memcpy(bp, pl, payload); /* ❌ attacker-controlled length */
  
+/* Fix added in OpenSSL 1.0.1g */
+if (1 + 2 + payload + 16 > s->s3->rrec.length) return 0;
① Attacker controls payload

The heartbeat message includes a length field. OpenSSL reads it directly — no validation. Attacker sets it to 0x4000 (16,384).

② Unchecked memcpy

Memory is allocated and memcpy copies payload bytes from the heap — not the actual received data length. Up to 64 KB of heap is sent back to the attacker.

③ The one-line fix

OpenSSL 1.0.1g added a single bounds check: if the claimed length exceeds the actual TLS record length, abort. That's all it took.
